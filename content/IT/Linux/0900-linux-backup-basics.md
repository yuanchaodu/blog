# Linux备份基础

Linux 备份是保障系统安全和业务连续性的基础工作。简单来说，备份就是把重要的数据复制到其他位置，以便在误删除、硬件故障、系统崩溃或遭受攻击时能够恢复。

## 一、为什么要备份

常见的数据丢失原因包括：

* 人为误操作
* 硬盘损坏
* 系统升级失败
* 病毒或勒索软件攻击
* 自然灾害或设备损坏

备份的目标是：

* 保护数据安全
* 缩短故障恢复时间
* 满足合规要求
* 保障业务连续运行

## 二、Linux备份对象

### 1. 用户数据

例如：

```bash
/home
/data
/opt/application/data
```

包括：

* 文档
* 图片
* 数据库文件
* 日志文件
* 上传文件

### 2. 系统配置

例如：

```bash
/etc
```

主要包含：

* 网络配置
* 用户配置
* 服务配置
* 安全策略

### 3. 应用程序

例如：

```bash
/opt
/usr/local
```

### 4. 数据库

常见数据库：

* MySQL
* MariaDB
* PostgreSQL
* Oracle

数据库通常需要逻辑备份或热备份，而不仅仅是复制数据文件。

## 三、常见备份类型

### 1. 完全备份（Full Backup）

每次备份全部数据。

示意：

```text
周日：全部数据
周一：全部数据
周二：全部数据
```

优点：

* 恢复简单

缺点：

* 占用空间大
* 时间长

---

### 2. 增量备份（Incremental Backup）

只备份自上次备份后发生变化的数据。

示意：

```text
周日：完全备份
周一：增量备份
周二：增量备份
周三：增量备份
```

恢复时：

```text
完全备份
+ 周一增量
+ 周二增量
+ 周三增量
```

优点：

* 节省存储空间
* 备份速度快

缺点：

* 恢复过程复杂

---

### 3. 差异备份（Differential Backup）

备份自上次完全备份后所有变化的数据。

示意：

```text
周日：完全备份
周一：差异备份
周二：差异备份
周三：差异备份
```

恢复时：

```text
完全备份
+ 最新差异备份
```

优点：

* 恢复较快

缺点：

* 备份文件越来越大

---

## 四、常见备份工具

### 1. cp

最简单的复制工具。

```bash
cp -r /data /backup/
```

适用于：

* 小规模数据
* 临时备份

---

### 2. tar

Linux最常用归档工具。

备份目录：

```bash
tar -czvf data.tar.gz /data
```

参数说明：

```text
-c 创建归档
-z gzip压缩
-v 显示过程
-f 指定文件名
```

恢复：

```bash
tar -xzvf data.tar.gz
```

---

### 3. rsync

Linux生产环境最常用备份工具。

备份：

```bash
rsync -av /data/ /backup/data/
```

参数：

```text
-a 保持权限属性
-v 显示过程
```

同步删除：

```bash
rsync -av --delete /data/ /backup/data/
```

特点：

* 增量同步
* 网络传输
* 效率高

---

### 4. scp

通过SSH远程备份。

```bash
scp data.tar.gz root@192.168.1.100:/backup
```

适用于：

* 小规模远程备份

---

### 5. rsync + SSH

企业环境最常见方案。

```bash
rsync -avz /data root@backup-server:/backup/data
```

优点：

* 安全传输
* 支持增量同步
* 节省带宽

## 五、数据库备份示例

### MySQL备份

逻辑备份：

```bash
mysqldump -uroot -p mydb > mydb.sql
```

压缩备份：

```bash
mysqldump -uroot -p mydb | gzip > mydb.sql.gz
```

恢复：

```bash
mysql -uroot -p mydb < mydb.sql
```

## 六、自动备份

Linux通常使用Cron实现定时备份。

编辑计划任务：

```bash
crontab -e
```

每天凌晨2点执行：

```bash
0 2 * * * /backup/backup.sh
```

查看任务：

```bash
crontab -l
```

## 七、备份策略（企业实践）

经典的“3-2-1原则”：

### 3份数据

```text
生产数据
本地备份
异地备份
```

### 2种介质

例如：

```text
磁盘阵列
对象存储
```

### 1份异地

例如：

```text
本地机房
云存储
```

这样即使机房发生故障，仍能恢复数据。

---

## 八、恢复比备份更重要

很多单位有备份，但从未验证恢复。

正确做法：

1. 定期执行备份
2. 定期检查备份日志
3. 定期进行恢复演练
4. 验证备份文件完整性
5. 记录恢复时间（RTO）
6. 记录允许丢失的数据时间（RPO）

例如：

```text
RTO = 2小时
表示故障后2小时内恢复业务

RPO = 15分钟
表示最多允许丢失15分钟数据
```

## 九、化工企业场景建议

结合工业企业和数字化工厂建设，建议采用分层备份策略：

| 对象             | 备份方式                    | 周期    |
| -------------- | ----------------------- | ----- |
| MES数据库         | MySQL/PostgreSQL逻辑备份+热备 | 每日    |
| Historian历史数据库 | 增量备份                    | 每日    |
| 文件服务器          | rsync同步                 | 每日    |
| 虚拟机            | 快照+整机备份                 | 每周    |
| 配置文件(/etc)     | tar归档                   | 每周    |
| PLC程序备份        | 专用软件导出                  | 变更后立即 |
| 网络设备配置         | 自动导出                    | 每周    |

推荐架构：

```text
生产服务器
      │
      ▼
本地备份服务器
      │
      ▼
异地灾备存储
      │
      ▼
云对象存储
```

对于您所在的化工企业IT环境，重点应关注 MES、ERP、质量管理系统、生产历史数据库（Historian）、虚拟化平台以及网络设备配置的备份，这些通常比操作系统本身更有价值。系统重装相对容易，而生产数据和工艺历史数据往往无法重新生成。
